# Contributing

This template is for UQRL industry projects, XR prototypes, games, interactive demos, and student team work. You can contribute ideas, bugs, setup problems, or asset issues by opening an issue.

You can contribute code by forking the repository, making a focused branch, and opening a pull request. For larger changes, create or claim an issue first and mention the issue ID in your pull request.

Need help with Git, branches, or pull requests? Ask the workshop director, project lead, or your UQRL project channel before guessing.

Please follow the [Code of Conduct](CODE_OF_CONDUCT.md) in all project spaces.

## Contribution Process

1. Find an issue to work on, or create one for the work you are about to do.
2. For industry projects, confirm priority and ownership with the workshop director before starting.
3. Create a branch with a short name, such as `feature/hand-tracking` or `fix/android-build`.
4. Make the smallest useful change and keep generated files out of the diff.
5. Update documentation before delivery. If setup, controls, assets, builds, or workflows changed, update `README.md`, relevant docs, or issue notes.
6. Open a pull request and list the engine/tool version, target platform, manual tests, and linked issues.
7. Request review from the workshop director, project lead, or assigned reviewer.
8. Merge only after approval and after generated cache/build output has been removed.

## Industry Project Rules

For industry projects, work distribution, priorities, scope changes, reviewer assignment, and delivery order are decided by the workshop director. Teams can suggest changes, but should not reassign work or change delivery scope without approval.

## Documentation

Every feature delivery should leave enough documentation for the next person to run, test, or review it. Prefer short notes that stay correct over long docs nobody maintains.
